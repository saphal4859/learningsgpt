What are beans in spring/springboot and its life cycle in detail in simple terms with code examples.

In Spring and Spring Boot, beans are objects that are instantiated, assembled, and managed by the Spring IoC (Inversion of Control) container. A bean can be any Java object that is configured in the Spring application context.

Key Points about Beans

  1. Managed by Spring: Beans are created, managed, and configured by the Spring container.
  2. Configuration: You can define beans using XML configuration, Java annotations, or Java configuration classes.
  3. Singleton by Default: Spring beans are singleton by default, meaning that a single instance of the bean is created and shared throughout the application. You can also define beans with other scopes like prototype, request, session, etc.

Bean Lifecycle

The lifecycle of a Spring bean can be divided into several stages:

  1. Instantiation: The Spring container instantiates the bean.
  2. Populate Properties: Spring sets the properties of the bean (Dependency Injection).
  3. Bean Name Aware: If the bean implements BeanNameAware, the Spring container calls setBeanName() method, passing the bean’s ID.
  4. Bean Factory Aware: If the bean implements BeanFactoryAware, Spring calls setBeanFactory() to provide the BeanFactory that created it.
  5. Application Context Aware: If the bean implements ApplicationContextAware, Spring calls setApplicationContext().
  6. Post-Processing: If the bean implements InitializingBean, the afterPropertiesSet() method is called. Additionally, any custom init methods defined are executed.
  7. Ready to Use: The bean is now ready for use.
  8. Destruction: When the application context is closed, if the bean implements DisposableBean, the destroy() method is called. Custom destroy methods are also executed.

Let's create a simple bean called MyBean:

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.beans.factory.annotation.PostConstruct;
import org.springframework.beans.factory.annotation.PreDestroy;
import org.springframework.stereotype.Component;

@Component
public class MyBean {

    private final Dependency dependency;

    @Autowired
    public MyBean(Dependency dependency) {
        this.dependency = dependency;
        System.out.println("MyBean instantiated!");
    }

    @PostConstruct
    public void init() {
        System.out.println("MyBean is initialized!");
    }

    public void doSomething() {
        System.out.println("Doing something with " + dependency.getName());
    }

    @PreDestroy
    public void cleanup() {
        System.out.println("MyBean is being destroyed!");
    }
}

Finally, create your main application class to run the Spring Boot application:

import org.springframework.boot.CommandLineRunner;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

Explanation of the Code

  3. @PostConstruct: This method is called after the bean's properties have been set, indicating that the bean is ready for use.
  4. @PreDestroy: This method is called just before the bean is destroyed, allowing for cleanup activities.

Explain about bean scopes in detail with code scenarios and examples

In Spring, the scope of a bean defines the lifecycle and visibility of that bean in the Spring container. Different scopes are used for different scenarios, depending on how you want to manage the lifecycle of your beans.

Bean Scopes in Spring

Spring provides several bean scopes, the most commonly used being:

  1. Singleton (default scope)
  2. Prototype
  3. Request
  4. Session
  5. Application
  6. Websocket

Let’s go through each scope in detail with code examples.

1. Singleton Scope

Definition: A singleton bean is created only once per Spring IoC container. All requests for that bean will return the same instance.

Code Example:

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.stereotype.Component;

@Component
class SingletonBean {
    public SingletonBean() {
        System.out.println("SingletonBean instance created!");
    }
    
    public void showMessage() {
        System.out.println("Hello from SingletonBean!");
    }
}

@Configuration
class AppConfig {
    @Bean
    public SingletonBean singletonBean() {
        return new SingletonBean();
    }
}

// Main Application Class
@SpringBootApplication
public class SingletonDemoApplication {

    public static void main(String[] args) {
        ApplicationContext context = SpringApplication.run(SingletonDemoApplication.class, args);
        
        SingletonBean bean1 = context.getBean(SingletonBean.class);
        bean1.showMessage();
        
        SingletonBean bean2 = context.getBean(SingletonBean.class);
        bean2.showMessage();
        
        System.out.println("Are both beans the same? " + (bean1 == bean2));
    }
}

Output:

SingletonBean instance created!
Hello from SingletonBean!
Hello from SingletonBean!
Are both beans the same? true

Explanation: The output shows that the SingletonBean instance is created only once, and both calls to getBean return the same instance.

1. Singleton Bean Behavior

In a singleton bean, if you modify the object's state in one class, it will reflect in all other classes that refer to the same instance. This is because all classes using the singleton bean reference the same object.

Code Example

Here's a simple example to illustrate this:

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.stereotype.Component;

@Component
class SingletonBean {
    private String message = "Initial Message";

    public String getMessage() {
        return message;
    }

    public void setMessage(String message) {
        this.message = message;
    }
}

@Component
class FirstClass {
    private final SingletonBean singletonBean;

    @Autowired
    public FirstClass(SingletonBean singletonBean) {
        this.singletonBean = singletonBean;
    }

    public void changeMessage() {
        singletonBean.setMessage("Changed from FirstClass");
    }
}

@Component
class SecondClass {
    private final SingletonBean singletonBean;

    @Autowired
    public SecondClass(SingletonBean singletonBean) {
        this.singletonBean = singletonBean;
    }

    public void printMessage() {
        System.out.println("Current Message: " + singletonBean.getMessage());
    }
}

@SpringBootApplication
public class SingletonExampleApplication {
    public static void main(String[] args) {
        var context = SpringApplication.run(SingletonExampleApplication.class, args);

        FirstClass firstClass = context.getBean(FirstClass.class);
        SecondClass secondClass = context.getBean(SecondClass.class);

        // Print initial message
        secondClass.printMessage(); // Outputs: Initial Message

        // Change message from FirstClass
        firstClass.changeMessage();

        // Print message again from SecondClass
        secondClass.printMessage(); // Outputs: Changed from FirstClass
    }
}

Explanation:

  1. SingletonBean: This bean has a message field that can be changed.
  2. FirstClass: This class modifies the message in the SingletonBean.
  3. SecondClass: This class prints the current message.
  4. In the main method, after calling changeMessage() from FirstClass, the message in SecondClass reflects that change because they both refer to the same instance of SingletonBean.

Summary

  • Specifies the scope of a bean (singleton, prototype, request, session, etc.).
  • Controls the lifecycle and visibility of beans managed by Spring.
  • Singleton Scope (@Scope("singleton"))
    • Default Scope: If you don't specify a scope, Spring uses singleton by default.
    • One Instance: Only one instance of the bean is created for the entire Spring container.
    • Shared: This single instance is shared across all requests and all classes that need it.
    • Example: Think of it as a single, shared object like a printer in an office. Everyone uses the same printer.
  • Prototype Scope (@Scope("prototype"))
    • Multiple Instances: A new instance of the bean is created every time it is requested. This means that every time we use @Autowired, a new object will be given.
    • Not Shared: Each request gets a new instance, so changes in one instance do not affect others.
    • Example: Similar to a personal pen, where each person gets their own pen and they are independent of each other.
  • Request Scope (@Scope("request"))
    • Web Applications: This scope is used in web applications, typically in controllers.
    • One Instance Per Request: A new instance of the bean is created for each HTTP request.
    • Limited Lifecycle: The bean exists only during the duration of the HTTP request.
    • Example: Imagine a temporary form on a website that you fill out. It exists only while you're interacting with that page.
      • 
@Controller
public class ProductController {
    @Autowired
    private ShoppingCart shoppingCart;
    // ...
}

@Scope("request") // Request scope for ShoppingCart
public class ShoppingCart {
    // ...
}
  • Session Scope (@Scope("session"))
    • Web Applications: Also used in web applications.
    • One Instance Per Session: A new instance of the bean is created for each HTTP session.
    • Shared in Session: The bean instance is shared across multiple requests within the same session.
    • Example: Like a shopping cart on an e-commerce site, which persists across multiple pages during your shopping session.
  • Global Session Scope (@Scope("globalSession"))
    • Portlets: This scope is specific to portlet-based web applications.
    • One Instance Per Global Session: Similar to session scope but tied to the global session across different portlets.
    • Example: In a portal environment where you have multiple portlets, this scope ensures the bean is shared across all portlets in the same session.
    • Lifetime: Until the session ends or is restarted, it will have the same object.
  • Application Scope (@Scope("application"))
    • Application-Wide: This scope creates a single instance of the bean for the entire lifecycle of the Spring application context.
    • Shared Across Application: The instance is shared across the entire application.
    • Example: Like a configuration setting that is available to the entire application for its entire lifetime.

10. Best Practices for Bean Scopes

Question: What are some best practices for managing bean scopes in a Spring application?

Answer:

  1. Use Singleton Scope for Shared Resources: For configuration classes or resources that do not maintain state between calls.
  2. Use Prototype Scope for Stateful Beans: For beans that need to maintain their own state.
  3. Avoid Using Prototype Beans in Singletons Directly: Use the @Lookup method or ApplicationContext to obtain instances.
  4. Consider Performance Implications: Creating new instances for prototype beans can affect performance; use wisely.
  5. Be Aware of Thread Safety: Singleton beans should be thread-safe if accessed from multiple threads.
  6. Use Request and Session Scopes for Web Applications: Leverage these scopes to maintain user state in web applications effectively.
